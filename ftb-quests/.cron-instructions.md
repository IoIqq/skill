目标
═══════════════════════════════════
从全球高人气 FTB 整合包中提炼「通用设计原子」——GoF 设计模式级别的可复用模式（名称 + 适用条件 + 实现要点 + 真实案例片段）。提取跨包通用的基础模式，不写整合包传记。

当前三硬伤（用户使用 skill 时遇到的问题）：
1. 物品跨级 — 任务要求当前阶段拿不到的物品
2. 顺序倒置 — 高级教学排在基础前面
3. 奖励断链 — reward 不引导下一步任务

布局三大缺口（Cycle 15+ 优先攻克）：
1. 碰撞检测不可执行 — 当前 topology-coordinates.md Phase 4 标注"LLM 不可执行"，AI 生成时遇到重叠只能加宽间距重来，没有修正策略。需要补一个可执行的碰撞修正算法：
   - 检测重叠：任意两节点距离 < min_distance（当前 1.0）
   - 修正优先级：hub（依赖数>=3）不动 > 中间节点微调 > 叶子节点大幅移动
   - 修正公式：沿依赖线方向平移，偏移量 = (min_distance - actual_distance) / 2 + 0.25
   - 整列平移：同一 y 行的所有节点一起移动，保持内部相对位置
   - 最多迭代 3 轮，仍重叠则增大全局 spacing 10% 重来
2. 大章混合拓扑无算法 — 200+ 任务的章节（如 ATM-10 create）包含多个子区域用不同拓扑，但没有分解算法。需要补：
   - 分解阈值：章节任务数 > 80 时触发子区域分解
   - 子区域识别：按依赖图的弱连通分量或模块度（modularity）切分
   - 子区域拓扑选择：每个子区域独立走 Phase 2 分类
   - 区域间过渡：用 bridge node（shape=hexagon, size=1.5）连接，坐标放在两区域中点
   - 区域布局策略：主区域占左侧 60% 宽度，子区域垂直排列在右侧
3. 非 expert 包校准数据稀疏 — highway_branch、diamond_convergence、tree_branching 各只有 1-2 个非 expert 案例。Cycle 15+ 应专门从 adventure/skyblock/farming/RPG 包中提取这些拓扑的校准数据，每个拓扑至少补到 5 个案例

  层1「完整布局算法」（伪代码级）：
    function layout_chapter(quests, dependencies, pack_type):
      1. 构建依赖图，计算每个节点的深度和扇入扇出
      2. 识别章节的主拓扑类型（线性/hub/并行/树/菱形/网格）
      3. 按拓扑类型分配初始坐标
      4. 碰撞检测 + 间距微调
      5. 按角色分配 shape/size/icon
      6. 返回每个 quest 的 {x, y, shape, size, icon}
    算法中每个步骤都给出可执行的伪代码，参数从真实数据中提取
  层2「约束规则」（B 层）：
    - 间距公式：spacing = f(quest_count, chapter_type)，给出上下界
    - shape 决策树：依赖数/角色/章节类型 → shape 选择
    - size 公式：size = f(角色, 依赖数)，给出 min/max
    - icon 规则：什么情况下设 icon、用什么 item
    - 碰撞检测规则：最小间距、重叠处理
    约束不是固定值而是公式+范围，让 AI 在约束内有发挥空间
  层3「真实案例坐标」：从研究包中提取的实际坐标数据，作为算法的校准基准

  从每个包的 chapter 文件中，自动识别并提取节点级拓扑模式。对每种拓扑，记录真实坐标数据：
  
  需要提取的 6 种拓扑：
  1. 线性链 — 连续依赖链的 x/y 递进规律（间距、方向、弯曲点）
  2. Hub + 扇形 — 中心节点与 N 个依赖的角度/距离分布（hub 的 shape/size、dep 的扇形半径和 y 范围）
  3. 并行多列 — 多条独立链的 x 间距、y 对齐方式
  4. 菱形汇聚 — 多路径汇入同一点的坐标收敛模式
  5. 树形分支 — hub 到子 hub 到叶子的层级间距（x-step 和 y-spread 随深度的变化）
  6. 网格/目录 — 无依赖的平铺排列（行列间距、换行规则）
  
  对每种拓扑，从真实数据中提取：
  - 实际 x/y 坐标序列（至少 3 个包的同类拓扑做对比）
  - shape 和 size 的分配规律（哪些节点用大图标、哪些用小图标）

  - 间距与节点数的关系：同一拓扑类型，5 个节点的间距 vs 20 个节点的间距 vs 50 个节点的间距（提取 spacing = f(N) 的经验公式）

  - size 与重要性的关系：capstone/毕业任务 size 分布 vs 普通中间步骤 size 分布
  - icon 设置率：每个包有多少百分比的 quest 设了 icon？设 icon 的 quest 有什么共同特征（如 hub、capstone、特定 shape）？

网络容错（fetch failed 处理）
═══════════════════════════════════

研究子 agent 重度依赖网页抓取（GitHub / MC百科 / CurseForge / Reddit 等），单个 fetch 失败不应浪费整晚。

处理规则：
1. 子 agent 报告某个包 fetch failed / 抓取超时 / 网络错误时，主 agent 不终止整轮：
   - 跳过当前包的抓取
   - 在 .research-lessons.md 记录：「<日期> <包名> fetch failed，跳过，待下轮重试」
   - 将该包追加到 .researched-packs.json5 的 discoveryQueue（标记 retry: true）
   - 继续下一个包
2. 连续 3 个不同的包都 fetch failed → 判定为网络异常（不是单个站点问题）：
   - 提前进入收尾流程（更新进度 + git commit + 同步副本 + 微信通知，标注"网络异常提前停止"）
   - 不再启动新的抓取子 agent
3. 同一个包连续 2 轮都 fetch failed → 在 lessons 中标注"该包源可能不可达"，降低其优先级，不再优先重试
4. 微信通知中报告：本轮跳过了哪些包（fetch failed）、是否因网络异常提前停止

注意：fetch failed 指网络层失败（DNS / 连接超时 / 5xx）。如果是 404（页面不存在）或内容解析失败，不算网络异常，按"该包无有效配置"处理，正常跳过即可。

知识浓缩规范（强制 — Cycle 18+）
═══════════════════════════════════

背景：reference 文件已膨胀到 300KB+，agent 无法在单次会话加载。所有新产出和存量内容必须按以下三层结构浓缩。

【三层存储结构】

1. 规则卡片（agent 生成时加载，必须精简）
   每条规则固定 6-8 行，格式：
   ### R42 — Stage-Internal Item Reachability
   触发: <什么情况下检查这条规则>
   检查: <具体检查什么，一行>
   严重度: <P0/P1/P2/INFO，是否阻断写入>
   修正: <违反时怎么修，一行>
   校准: <N 包验证（包名列表），参数范围（如有）>

   禁止在卡片里写：完整推理过程、内嵌案例分析、边界情况长篇讨论。这些进证据日志。

2. 校准表（拓扑案例，agent 按需加载）
   每个案例压缩成一行：
   | Case | Pack | Chapter | Quests | Topology | Key params | Finding |
   | 45 | MC-Odyssey-3 | exploration | 12 | parallel_columns | 1.5×2.5 spacing | diamond=类别标记 |

   校准表末尾附按拓扑类型汇总的参数范围：
   hub_fan: hub_radius 3.5–8.0（Cases 12/23/34/50）| fan spacing 1.0–1.5

3. 证据日志（夜间任务写入，agent 不主动加载）
   位置：reference/design/evidence/cycle-<N>-<topic>.md
   内容：原始坐标数据、完整审计发现、案例详细分析、被压缩掉的长篇内容。
   只有 agent 需要校准某个具体参数时才查证据日志。

【写入规则 — 每个子 agent 必须遵守】

1. 新发现先判断："这改变了某条规则的参数边界/触发条件，还是只是又一个确认实例？"
2. 只是确认实例 → 写证据日志 + 在规则卡片"校准"行追加包名。不动卡片正文。
3. 改变参数边界或发现新触发条件 → 更新卡片对应字段（仍保持 6-8 行）。
4. 全新规则 → 写一张新卡片 + 证据日志记原始数据。
5. 每个拓扑类型的校准案例超过 5 个后，新案例只记入证据日志，不再加进校准表（边际收益递减）。
6. 全新案例 → 校准表加一行 + 证据日志记完整数据。

【体积预算】

- progression-rules.md（规则卡片）≤ 60KB
- topology-algorithms.md（算法+公式）≤ 60KB
- topology-calibration.md（校准表）≤ 25KB
- micro-patterns.md（模式卡片）≤ 50KB
- anti-patterns.md（反模式卡片）≤ 45KB
- 证据日志：不限（不加载进上下文）

超出预算时：检查是否有规则可以合并、低置信度规则移入 drafts/candidates.md、冗余案例移入证据日志。

【存量压缩 — 分批执行】

当前 progression-rules.md（303KB）、topology-coordinates.md（309KB）、micro-patterns.md（164KB）、anti-patterns.md（129KB）包含大量内嵌案例，需分批压缩。每轮 cycle 的【阶段 0】压缩一个文件，顺序：
  第 1 轮: topology-coordinates.md → 拆为 topology-algorithms.md + topology-calibration.md
  第 2 轮: progression-rules.md → 重写为卡片格式，案例移入 evidence/
  第 3 轮: micro-patterns.md → 卡片化
  第 4 轮: anti-patterns.md → 卡片化

压缩安全规则：
- 压缩前必须完整读取原文件，确认每条规则/案例的核心信息（触发/检查/严重度/修正/参数）无遗漏后再写入新格式
- 压缩产生的删除属于高风险改动：原文件备份到 drafts/pre-compress-<文件名>-<日期>.md 后再 commit
- 每轮只压缩一个文件，commit + push + 同步副本，不批量重构
- 压缩完成后跑一遍引用自检：确认 SKILL.md 路由表和 module-index.md 指向的文件路径仍然有效（如 topology-coordinates.md 拆分后需同步更新路由表为 topology-algorithms.md / topology-calibration.md）

完成后通知（必须执行）
═══════════════════════════════════
每次执行完毕（无论正常完成还是时间截止），将摘要发送到「我的微信」。

正常完成格式：
【ftb-quests 夜间研究 · 周期 X · 阶段 Y/5】
- 今晚完成：具体工作内容
- 新研究包：本阶段新研究的包名列表（共累计 N 个）
- 主要发现：2-3 条关键发现
- Git 提交：commit hash + 提交信息
- 已同步副本：1/2/3（标注哪些成功）
- 已直接修改：文件列表
- 待审核草稿：文件列表 + 理由（高风险）
- 经验库新增：本阶段关键教训
- 下一步计划：下次要做什么 + 下轮主力渠道

时间窗口截止格式：
【ftb-quests 夜间研究 · 时间截止 · 周期 X · 阶段 Y】
- 停止原因：时间窗口截止
- 本次完成：已完成的阶段
- 未完成：停在哪个阶段
- Git 提交：commit hash
- 已同步副本：1/2/3
- 下次继续：从阶段 Y 继续

整轮循环完成格式：
【ftb-quests 研究 · 第 X 轮完成】
- 本轮主题
- 本轮新研究包：包名列表（累计总数 N 个）
- 新增或更新模块
- Git 提交记录
- 已同步副本：1/2/3
- 下轮主力渠道 + 计划
