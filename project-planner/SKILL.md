# Project Planner — 项目技术规划

## 概述

这是一个项目启动阶段的技术规划 skill。当用户给出一个项目需求描述时，按照以下 5 个步骤完成从需求拆解到完整技术方案的规划输出。本 skill **不替用户做决策**，而是提供经过调研和评估的选项与建议，由用户最终拍板。

## 核心原则

- **保持中立客观**：对每个方案如实记录其优缺点、许可证类型、社区活跃度等客观信息，不预设立场。
- **基于事实比较**：方案筛选依据可查证的信息（许可证条款、GitHub Stars、最近更新日期、文档完善程度等）。
- **透明可追溯**：对每个推荐方案说明选择理由，并注明潜在风险。
- **输出即交付**：最终成果是一份结构化的技术规划文档，可直接作为后续开发的基础参考。

## 工作流程

### 步骤 1：需求分析与模块拆解

接收用户输入的项目需求，进行以下处理：

1. **理解核心需求**：读用户的项目描述，必要时通过提问澄清模糊点（目标用户、核心功能边界、技术约束等）。
2. **功能模块拆解**：将项目拆解为 3~8 个独立的功能模块。常见的模块维度包括：
   - 前端展示层（用户界面、交互）
   - 后端服务层（API、业务逻辑）
   - 数据存储层（数据库、缓存、文件存储）
   - 身份认证与权限管理
   - AI/模型集成（如果涉及）
   - 部署与运维
   - 第三方服务集成
   - 实时通信（如果涉及）
3. **记录每个模块的关键技术需求与约束**：技术栈偏好（如 React/Vue/Python/Go）、性能要求、安全要求、扩展性要求等。
4. **将拆解结果展示给用户确认**，根据用户反馈调整。

### 步骤 2：联网搜索免费开源方案

针对每个功能模块，使用 WebSearch 工具搜索是否存在现成的免费开源方案。搜索策略：

1. **搜索范围**：
   - 开源项目（GitHub、GitLab、Gitee 等）
   - 免费 API 服务（有充足的免费额度）
   - 免费 SDK / 库
   - 免费托管服务（如 Vercel Free Tier、Supabase Free Tier 等）
2. **搜索关键词示例**（以"在线协作白板"为例）：
   - `best open source collaborative whiteboard`
   - `free online whiteboard API`
   - `开源 在线白板 协作 方案`
   - 针对不同模块使用不同关键词
3. **对每个找到的方案，记录以下信息**：
   - **名称**：方案/项目的名称
   - **功能描述**：一句话说明它能做什么
   - **许可证类型**：MIT、Apache 2.0、GPL、AGPL、BSL 等
   - **是否完全免费**：有无隐藏费用或使用限制（如自托管免费但云服务收费）
   - **集成难度**：容易 / 中等 / 困难
   - **项目活跃度**：GitHub Stars、最近更新时间、Issue 响应速度
   - **官方文档质量**：好 / 一般 / 差
   - **相关内容链接**：仓库地址、官网等

### 步骤 3：方案评估与筛选

对收集到的方案按以下标准进行对比和筛选：

1. **免费性核查**（首要标准）：
   - 是否真正免费？有无隐藏费用？
   - 免费额度是否足以支撑项目初期的使用？
   - 自部署版本是否与付费版本功能一致？
2. **许可证合规**：
   - 是否允许商业使用？
   - 是否需要开源衍生代码？（如 GPL/AGPL 的传染性）
   - 有无额外的条款限制？
3. **社区活跃度与维护状态**：
   - GitHub Stars / Forks 数量
   - 最近一次提交时间（理想情况：过去 3 个月内有活跃提交）
   - Issue 和 PR 的响应情况
   - 是否有活跃的社区（Discord、Slack、论坛等）
4. **兼容性评估**：
   - 与其他模块所选方案的技术栈是否兼容？
   - 是否有已知的集成冲突？
   - 依赖的运行时版本是否一致？
5. **集成复杂度**：
   - 需要多少开发工作才能集成？
   - 是否有现成的示例或模板可用？
   - 学习曲线如何？

**筛选结果呈现格式**：对每个模块，以表格形式列出 2~3 个候选方案，标明各自的优劣势，给出推荐方案和选择理由，并说明未被选中的方案的落选原因。

### 步骤 4：组合规划

将各模块筛选后的方案整合为完整的项目架构方案，输出内容应包括：

1. **项目整体架构描述**：用文字描述各模块如何组织在一起。
2. **各模块所选方案及其选择理由**：简要回顾每个模块的最终选择。
3. **模块间集成方式与数据流向**：描述模块之间的接口、通信协议、数据格式等。
4. **技术风险与替代方案**：
   - 每个所选方案的潜在风险（如许可证变更、项目停止维护等）
   - 针对每个风险的备选方案建议
5. **项目实施步骤建议**：推荐的分阶段实施计划，从 MVP 到完整功能的分阶段规划。

### 步骤 5：输出规划文档

以前一步的组合规划为基础，输出一份结构化的规划文档，格式为 Markdown 文件，文件命名规则为 `项目名称-技术规划方案.md`，保存在当前工作目录下。


## 注意事项

1. **搜索要全面**：用中英文关键词分别搜索，不要遗漏优质的中文开源项目。
2. **如实记录**：对于不确定的信息（如许可证细节），应说明"需进一步确认"而非猜测。
3. **不要过度承诺**：对实施时间的估计应留有余地，说明"取决于团队规模和经验"。
4. **保持交互**：在关键节点（如模块拆解后、方案推荐后）与用户确认，避免方向偏差。
5. **文档交付**：最终文档应格式规整、信息完整，可直接用于开发团队的讨论和决策参考。
